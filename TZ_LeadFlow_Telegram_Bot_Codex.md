# Техническое задание на Telegram-бот LeadFlow

Формат: компактное ТЗ для реализации через Codex.

---

## 1. Цель бота

Telegram-бот LeadFlow нужен для первичного прогрева пользователей, выдачи бесплатного PDF-гайда, перевода в канал и сбора заявок на бесплатный разбор бизнес-процессов.

Главное действие бота: привести пользователя к кнопке **«Получить разбор»** и корректно отправить заявку в админ-чат.

---

## 2. Что должно быть реализовано

- Стартовый экран с обложкой, приветственным текстом и inline-кнопками.
- Раздел **«Получить разбор»** с изображением, инструкцией, сбором заявки и отправкой уведомления в админ-чат.
- Раздел **«Забрать гайд»** с отправкой PDF-файла без предварительного сбора контакта.
- Раздел **«Канал с разборами»** с описанием и кнопкой перехода в Telegram-канал.
- Раздел **«О нас»** с изображением, текстом о LeadFlow и кнопками перехода.
- Раздел **«Написать нам»** с контактами Ростислава и Даниила.
- Fallback-сценарии: непонятный текст, короткая заявка, отсутствие контакта, не текстовые сообщения, возврат в меню.

---

## 3. Что не нужно реализовывать в первой версии

- Не делать CRM, личный кабинет, оплату, роли пользователей и админ-панель.
- Не делать автоматическую генерацию AI-аудита внутри бота. Аудит готовится вручную.
- Не закрывать PDF-гайд за заявкой или подпиской.
- Не добавлять Google Sheets, если это отдельно не будет согласовано.
- Не использовать постоянную нижнюю клавиатуру. Нужны inline-кнопки под сообщениями.

---

## 4. Рекомендуемый стек

Codex должен реализовать бота как обычное приложение Telegram Bot API.

Рекомендуемый стек:

- Python 3.11+
- aiogram 3.x
- FSM для состояний пользователя
- переменные окружения для `BOT_TOKEN` и `ADMIN_CHAT_ID`
- отдельный конфиг для текстов, ссылок и путей к файлам

Если в проекте уже используется другой стек, логику можно перенести без изменения сценариев.

## 6. Важная пометка про изображения и PDF

Все изображения и PDF должны лежать в папке `assets/`.

Рекомендуемая структура:

```text
assets/
├── 01_start_cover.png
├── 02_audit_cover.png
├── 03_about_cover.png
└── LeadFlow_AI_Guide_2026.pdf
```

### Назначение файлов

| Файл | Где используется |
|---|---|
| `01_start_cover.png` | изображение при команде `/start` |
| `02_audit_cover.png` | изображение в разделе **«Получить разбор»** |
| `03_about_cover.png` | изображение в разделе **«О нас»** |
| `LeadFlow_AI_Guide_2026.pdf` | PDF-файл в разделе **«Забрать гайд»** |

### Где изображения отправляются

Изображения отправляются только в этих сценариях:

- `/start`
- **«Получить разбор»**
- **«О нас»**

Изображения не отправляются в этих разделах:

- **«Забрать гайд»**
- **«Канал с разборами»**
- **«Написать нам»**

### Требования к именам файлов

- Не использовать кириллицу в названиях файлов.
- Не использовать пробелы.
- Не использовать названия вроде `финал2.png`, `картинка новая.png`, `photo_123.png`.
- Использовать точные имена из этого ТЗ, чтобы Codex мог однозначно подставить файлы в код.

### Пример конфига для Codex

```python
START_IMAGE = "assets/01_start_cover.png"
AUDIT_IMAGE = "assets/02_audit_cover.png"
ABOUT_IMAGE = "assets/03_about_cover.png"
GUIDE_FILE = "assets/LeadFlow_AI_Guide_2026.pdf"
```

Если файл изображения или PDF отсутствует, бот не должен падать. Ошибка должна логироваться, а пользователю должен показываться fallback-текст.

## 10. Раздел «Получить разбор»

Триггер: `callback_data = audit_start`.

Бот отправляет изображение `assets/02_audit_cover.png`, затем текст-инструкцию и переводит пользователя в состояние `WAITING_AUDIT_DESCRIPTION`.

### Текст раздела

```text
Бесплатный разбор процессов

Опишите ваш бизнес одним сообщением — мы посмотрим, где сейчас может теряться время, заявки или деньги, покажем, где AI и автоматизация могут дать практическую пользу, и предложим 3 процесса, которые можно автоматизировать первыми.

Что лучше указать:

— чем занимается бизнес;
— откуда приходят заявки;
— что сейчас делается вручную;
— что хотите улучшить;
— контакт для обратной связи.

Пример:

“Мебель под заказ. Заявки идут с Avito и Telegram. Менеджер вручную отвечает про цены, размеры и сроки. Хочется быстрее обрабатывать заявки и не терять клиентов. Контакт: @username”

После этого мы посмотрим задачу и отправим вам первичный разбор.
```

### Кнопки до отправки заявки

| Кнопка | Действие |
|---|---|
| Посмотреть разборы в канале | `callback_data = channel` |
| Назад в меню | `callback_data = main_menu` |

### Проверки заявки

- Если пользователь отправил не текст, бот просит отправить описание обычным текстовым сообщением.
- Если текст короче 30 символов, бот просит описать бизнес подробнее.
- Если в тексте есть контакт, бот использует его.
- Если в тексте нет контакта, но у пользователя есть Telegram username, бот использует username как контакт.
- Если контакта нет и Telegram username отсутствует, бот просит контакт отдельным сообщением и переводит пользователя в `WAITING_CONTACT`.

### Ответ, если описание слишком короткое

```text
Нужно чуть подробнее, иначе мы не сможем сделать нормальный разбор.

Напишите одним сообщением:

— чем занимается бизнес;
— откуда приходят заявки;
— что сейчас делается вручную;
— что хотите улучшить;
— контакт для обратной связи.

Пример:
“Мебель под заказ. Заявки идут с Avito и Telegram. Менеджер вручную отвечает клиентам вручную. Хочется быстрее обрабатывать заявки. Контакт: @username”
```

### Ответ, если нет контакта

```text
Описание получили. Оставьте, пожалуйста, контакт для обратной связи: Telegram, телефон или email.

Можно просто отправить @username или номер телефона.
```

### Подтверждение после успешной заявки

```text
Спасибо, заявку получили.

Мы посмотрим ваш бизнес-процесс и подготовим короткий аудит:

— с какого MVP лучше начать.

Ответ отправим по указанному контакту.
```

## 16. Fallback-сценарии

| Ситуация | Действие бота |
|---|---|
| Текст вне активного сценария | показать главное меню |
| `/start` в любом состоянии | сбросить состояние и запустить стартовый сценарий заново |
| Назад в меню | сбросить состояние, показать меню без обложки |
| Во время заявки отправлено не текстовое сообщение | попросить отправить описание текстом |
| Описание короче 30 символов | попросить описать подробнее |
| Нет контакта, но есть Telegram username | использовать username как контакт |
| Нет контакта и нет username | попросить контакт отдельным сообщением |
| Бот ждёт контакт, пользователь отправил неподходящее сообщение | повторно попросить Telegram, телефон или email |
| PDF временно недоступен | предложить написать напрямую или получить разбор |
| Заявка не ушла в админ-чат | залогировать ошибку, пользователю не показывать технические детали |

## 18. Требования к структуре кода

- Для inline-кнопок использовать единые `callback_data` из этого ТЗ.

- Ошибки отправки в админ-чат логировать.
- Проект должен запускаться локально командой из README.

- Команда `/start` показывает обложку, текст и главное меню.
- Все inline-кнопки открывают нужные разделы.

- Кнопка **«Назад в меню»** возвращает в меню без повторной обложки.
- Непонятные сообщения не ломают сценарий, бот возвращает пользователя к меню.
- Токен бота не хранится в коде.
